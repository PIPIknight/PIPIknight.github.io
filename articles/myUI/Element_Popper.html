Element Popper #

​ UI设计了一个这周新的 区号 + 手机 输入框，本想着伟大的element plus中终于也更新了复合型输入框：

html
<el-input v-model="input1" placeholder="Please input">
    <template #prepend>Http://</template>
</el-input>

然而设计要求，区号的选择框下拉选项要和整体输入框一致。本想着好嘛，不就是改改样式。然而不管怎么改，样式穿透都无效，查看页面元素才发现，下拉框全部位于独立于 id=‘app’ 的div盒子的另一个 ‘el-popper-container’ 容器里：

html
<body>
    <div id='app' data-v-app>
        ……
    </div>
    <script type="module" src="/src/main.ts"></script>
    <div id='el-popper-container-7045'>
        ……
    </div>
</body>

Element UI并没有相关文档，所以百度了一下资料，学习学习，内容对于Element Plus依旧适用，因为 Element 的核心原理没有大的变化。

以下主要参考 https://zhuanlan.zhihu.com/p/450317200

一、ElementUI 里所有弹出层的两种模式 #

​ ElementUI 的弹出层（包括但不限于：dialog, select, popover, date-picker 等）在元素定位上，都有两种实现方式，分别是：

  • 方案一： append-to-body 式。此模式下，弹出层会被放在 body 元素上，通过 position:fixed 定位，配合动态的 top 和 left 属性，完成弹出元素的定位。

  • 方案二： 非 append-to-body 式。此模式下，弹出层通过 position:absolute 定位，配合其父元素 position:relative 来完成弹出元素的定位。

在大多数情况下， ElementUI 都是默认使用的 『方案一：append-to-body 式』。

原因很简单，因为『方案二： 非 append-to-body 式』 存在严重副作用，只有迫不得已的情况下才需要使用。

例如，当弹出层组件的父元素拥有 position: relative; overflow: auto 样式时，是否 append-to-body 可能直接影响组件的显示（ 比如 display:flex 会使其父子的 相对定位出现问题，父定位要求等原因 ），我的理解是 非 append-to-body 模式，对父级存在依赖，MDN对绝对定位的描述：绝对定位元素会被移出正常文档流，并不为元素预留空间，通过指定元素相对于 最近的非 static 定位祖先元素 的偏移，来确定元素位置。相对之下，append-to-body模式仅针对于body元素，稳定。

『非 append-to-body 模式』会在多种场景下导致问题。

这也是为什么 ElementUI 会把所有弹出层设置为『append-to-body 模式』的根本原因？

二、ElementUI 是如何管理弹出层的z-index的？ #

第一节已经聊到了，除非特别指定，ElementUI 都会自动使用 『append-to-body 模式』来进行弹出层的实现。 在这个前提下，不妨假象自己是 ElementUI 的开发人员，你会怎么实现以下效果：

应该如何保证 dialog 组件里，select 组件的弹出层高于 dialog 的弹出层？ 应该如何保证 dialog-1 上再次弹出的 dialog-2 的层级高于 dialog-1的层级？ ElementUI 项目组的人员在设计这块的时候，想出了一个简单却让人拍案叫绝的办法：只要让新出现的弹出层，永远比之前所有弹出层的层级要高，就不会有『新弹层』被『旧弹层』遮盖的事情发生。

假设一个场景：“弹窗-1”上点击某个按钮弹出“弹窗-2”，而“弹窗-2”上又有个“下拉框组件”，此时它们的 z-index 会是什么样的呢？

如上图所示，只要保证每次“弹出层”在弹出时，都在当前最大 z-index 的基础上 +1，就可以保证没有弹出层会被无故挡住。

记住这个机制，它是 ElementUI 弹出层的核心实现机制。实现这个核心机制的便是 —— PopupManager

三、PopupManager ( vue3 中改为 useZIndex ) #

PopupManager ( vue3 中改为 useZIndex ): 为弹出层提供获取实例、注册、注销 等各种能力，但其最重要的能力，是提供了 z-index 的层级管理能力，使用示例：

vue
<template>
  <div>
    <el-button @click="onClick">增加</el-button>
    z-index: {{ value }}
  </div>
</template>
<script lang="ts">
import { useZIndex } from "element-plus";
import { ref } from "vue";
export default {
  name: "demo",
  setup() {
    let value = ref(0);
    const onClick = () => {
      const { nextZIndex } = useZIndex();
      value.value = nextZIndex();
    };
    return {
      value,
      onClick,
    };
  },
};
</script>

第一次 useZIndex.nextZIndex() 会返回2000，然后每次点击都会 +1。

后续的 +1 很好理解，每次弹出层要获取到的最新可用 z-index 就是要比之前的那一层多 1。

但是为什么一开始会是 2000 呢？

其实这是 ElementUI 内置的一个“弹出层 z-index 基数”，但它是可以进行修改的。

在安装 ElementUI 时，通过以下代码可以让内置的 2000 变成 3000。

js
// 这可以让弹出层的 z-index 从 3000 开始递增
Vue.use(Element, { zIndex: 3000 });

四、全屏问题 #

浏览器默认全屏api

js
Element.requestFullscreen()

但是，当全屏时，全屏弹窗就显得难以实现，因为：“官方全屏是设定层面的高于一切，那些 append-to-body的弹窗，无论 z-index 再高，也绝对不可能被显示出来”。如此一来，所有 非 append-to-body 的弹出层就全部GG了。

以下是demo组件：

vue
<template>
  <div
    :class="{ 'my-full-screen': isFullScreen }"
    :style="{ zIndex: currentZIndex }"
  >
    <slot></slot>
  </div>
</template>
<script lang="ts">
import { useZIndex } from "element-plus";
import { ref } from "vue";
export default {
  name: "demo",
  setup() {
    let isFullScreen = ref(false);
    let currentZIndex = ref(0);

    const request = () => {
      isFullScreen.value = true;
      const { nextZIndex } = useZIndex();
      currentZIndex.value = nextZIndex();
    };

    return {
      currentZIndex,
      isFullScreen,
      request
    };
  },
};
</script>
<style lang="scss" scoped>
.my-full-screen {
  position: fixed !important;
  top: 0 !important;
  left: 0 !important;
  right: 0 !important;
  bottom: 0 !important;
  width: 100% !important;
  height: 100% !important;
}
</style>

以下是使用，及其效果：

vue
<template>
  <div class="like-a-page">
    <FullScreen class="like-a-module" ref="myRef">
      <el-button @click="requestOwnFullScreen">我封装的全屏</el-button>
      <el-select>
        <el-option>随便写点啥</el-option>
      </el-select>
    </FullScreen>
    <div class="like-a-module" ref="browserRef">
      <el-button @click="requestBrowserFullScreen">官方的全屏</el-button>
      <el-select>
        <el-option>随便写点啥</el-option>
      </el-select>
    </div>
  </div>
</template>
<script lang="ts">
import FullScreen from "./index.vue";
import { onMounted, ref } from "vue";
export default {
  name: "demo2",
  components: {
    FullScreen,
  },
  setup() {
    const browserRef = ref(null);
    const myRef = ref(null);

    const requestBrowserFullScreen = () => {
      browserRef.value.requestFullscreen();
    };
    const requestOwnFullScreen = () => {
      myRef.value.request();
    };
    return {
      browserRef,
      myRef,
      requestBrowserFullScreen,
      requestOwnFullScreen
    };
  },
};
</script>
<style lang="scss" scoped>
.like-a-page {
  width: 800px;
  height: 600px;
  background-color: #e5e5e5;
  display: flex;
  flex-direction: column;
  justify-content: space-around;
  align-items: center;
}
.like-a-module {
  width: 400px;
  height: 200px;
  background-color: white;
}
</style>

可自行拷贝下来对比，效果还算OK！

五、认识：万能弹出组件（vue-popper） #

我们所能接触到的，ElementUI 中的大部分弹出层都是基于 vue-popper 组件来实现的。

简单罗列一下：select、date-picker族、级联cascader、dropdown、popover、tooltip...等等，这些组件都是基于 vue-popper 组件来实现弹出层的。
